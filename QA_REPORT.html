What does this mean?
Every button, every filter, every export, every page in the application was tested automatically — not by clicking around manually, but by a script that simulates exactly what a real user does: logging in, selecting a database, generating reports with different filter combinations, exporting to Excel/CSV/PDF, opening print previews, scheduling reports, sending emails, and verifying that all data is returned correctly. If any single endpoint fails or returns unexpected data, the test flags it immediately.

Verification Criteria Per Test

Page Load Tests
HTTP 200 status + response contains expected HTML elements (form IDs, button labels, table headers specific to each report)
Data Generation Tests
POST with valid filter parameters → HTTP 200 + response body contains data rows or valid empty result (no error HTML)
Export Tests
GET with filter params → HTTP 200 + Content-Type matches format (xlsx/csv/pdf) + file size > minimum threshold
API Tests
GET/POST → HTTP 200 + valid JSON response (array or object). Empty arrays ([]) accepted when no data exists in demo DB

Session Management

Each test script maintains a persistent WebRequestSession object that carries authentication cookies across all requests — exactly as a browser would. The flow mirrors real user behavior: Login → Select Database → Connect → Use Application → Disconnect. Anti-forgery tokens are extracted from form HTML and submitted with POST requests, validating that CSRF protection works correctly throughout.

🐛 Issues Discovered During Verification

The verification process uncovered 4 bugs. All were fixed and deployed before the report was finalized.
Purchases vs Sales — Export Staleness Bug CRITICAL
When a user changed filters after generating the report, all exports (Excel, CSV, PDF), Print Preview, Send Email, and AI Analysis continued using the original filter values from page load — not the current selections. This meant exported data could differ from what was displayed on screen.
✓ FIXED — Centralized collectPsParams() helper reads live DOM values for all 6 affected functions.
Scheduled AI Analysis — Wrong Report Data CRITICAL
The scheduler's AI analysis pipeline only had handling for Average Basket and Purchases vs Sales. When running AI analysis for Profit & Loss, Trial Balance, or Catalogue, it silently generated Average Basket CSV data instead — causing the AI to analyze completely wrong numbers and send misleading insights via email.
✓ FIXED — Added explicit filter construction and CSV generation for P&L, Trial Balance, and Catalogue in ScheduleExecutionService.
Charts — Missing Print Preview Button MEDIUM
The Charts report had a working Print Preview controller action on the server, but no UI button to trigger it. Every other report had a Print/Preview option.
✓ FIXED — Added Preview toggle button with inline iframe display (no new tab needed).
Below Min Stock — Missing Server-Side Exports MEDIUM
BMS only had a basic client-side CSV export. No Excel, no PDF, and the CSV was generated in the browser rather than on the server like all other reports.
✓ FIXED — Added three server-side export endpoints (Excel with color-coding, CSV, PDF in landscape) matching the pattern used by all other reports.
